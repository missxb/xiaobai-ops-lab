# 企业级日志收集分析平台

> 基于 ELK Stack + Kafka + Filebeat 的企业级日志平台
> 参考方案：美团日志平台、滴滴 DLog、阿里云 SLS

---

## 一、架构设计

### 1.2 容量规划

| 组件 | 配置 | 说明 |
|------|------|------|
| Kafka | 3 节点, 16C32G, 500GB SSD | 日志缓冲 |
| Logstash | 4 节点, 8C16G | 日志处理 |
| Elasticsearch | 3 节点, 16C64G, 1TB SSD | 日志存储 |
| Kibana | 2 节点, 8C16G | 可视化 |

### 1.3 日志保留策略

| 日志类型 | 保留时间 | 存储策略 |
|---------|---------|---------|
| 应用日志 | 30 天 | Hot→Warm→Delete |
| 访问日志 | 90 天 | Hot→Warm→Cold→Delete |
| 审计日志 | 365 天 | Hot→Warm→Cold→Archive |
| 系统日志 | 30 天 | Hot→Warm→Delete |

### 2.2 Topic 配置

```yaml
# kafka-topics-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: kafka-topics
  namespace: logging
data:
  create-topics.sh: |
    #!/bin/bash
    # 创建日志 Topic
    
    KAFKA_BROKERS="kafka-0.kafka:9092,kafka-1.kafka:9092,kafka-2.kafka:9092"
    
    # 应用日志
    kafka-topics.sh --bootstrap-server ${KAFKA_BROKERS} \
      --create --if-not-exists \
      --topic app-logs \
      --partitions 12 \
      --replication-factor 3 \
      --config retention.ms=2592000000 \
      --config max.bytes.per.partition=104857600 \
      --config segment.ms=3600000
    
    # 访问日志
    kafka-topics.sh --bootstrap-server ${KAFKA_BROKERS} \
      --create --if-not-exists \
      --topic access-logs \
      --partitions 12 \
      --replication-factor 3 \
      --config retention.ms=7776000000 \
      --config max.bytes.per.partition=104857600 \
      --config segment.ms=3600000
    
    # 系统日志
    kafka-topics.sh --bootstrap-server ${KAFKA_BROKERS} \
      --create --if-not-exists \
      --topic system-logs \
      --partitions 6 \
      --replication-factor 3 \
      --config retention.ms=2592000000 \
      --config max.bytes.per.partition=104857600 \
      --config segment.ms=3600000
    
    # 审计日志
    kafka-topics.sh --bootstrap-server ${KAFKA_BROKERS} \
      --create --if-not-exists \
      --topic audit-logs \
      --partitions 6 \
      --replication-factor 3 \
      --config retention.ms=31536000000 \
      --config max.bytes.per.partition=104857600 \
      --config segment.ms=3600000
    
    echo "=== Topic 创建完成 ==="
    kafka-topics.sh --bootstrap-server ${KAFKA_BROKERS} --list

---

            - name: scripts
```

## 四、Logstash 配置

### 4.1 完整 Pipeline 配置

```ruby
# logstash.conf - Logstash 主配置
input {
  # 从 Kafka 读取应用日志
  kafka {
    bootstrap_servers => "kafka-0.kafka:9092,kafka-1.kafka:9092,kafka-2.kafka:9092"
    topics => ["app-logs"]
    group_id => "logstash-app-logs"
    consumer_threads => 4
    decorate_events => true
    codec => json
    auto_offset_reset => "latest"
    max_poll_interval => 300000
    session_timeout_ms => 30000
  }
  
  # 从 Kafka 读取访问日志
  kafka {
    bootstrap_servers => "kafka-0.kafka:9092,kafka-1.kafka:9092,kafka-2.kafka:9092"
    topics => ["access-logs"]
    group_id => "logstash-access-logs"
    consumer_threads => 4
    decorate_events => true
    codec => json
    auto_offset_reset => "latest"
  }
  
  # 从 Kafka 读取系统日志
  kafka {
    bootstrap_servers => "kafka-0.kafka:9092,kafka-1.kafka:9092,kafka-2.kafka:9092"
    topics => ["system-logs"]
    group_id => "logstash-system-logs"
    consumer_threads => 2
    decorate_events => true
    codec => json
    auto_offset_reset => "latest"
  }
  
  # 从 Kafka 读取审计日志
  kafka {
    bootstrap_servers => "kafka-0.kafka:9092,kafka-1.kafka:9092,kafka-2.kafka:9092"
    topics => ["audit-logs"]
    group_id => "logstash-audit-logs"
    consumer_threads => 2
    decorate_events => true
    codec => json
    auto_offset_reset => "latest"
  }
}

filter {
  # ===== 通用处理 =====
  # 添加处理时间
  ruby {
    code => "event.set('logstash_processed_at', Time.now.utc.iso8601)"
  }
  
  # 移除不需要的字段
  mutate {
    remove_field => ["host", "@version"]
  }
  
  # ===== 应用日志处理 =====
  if [topic] == "app-logs" {
    # 解析日志级别
    grok {
      match => { "message" => "%{LOGLEVEL:log_level}" }
      tag_on_failure => ["_grokparsefailure_app"]
    }
    
    # 提取 trace_id
    grok {
      match => { "message" => "trace_id=%{DATA:trace_id}" }
      tag_on_failure => ["_grokparsefailure_trace"]
    }
    
    # 提取请求信息
    grok {
      match => { "message" => "method=%{WORD:method} url=%{DATA:url} status=%{INT:status} duration=%{INT:duration}" }
      tag_on_failure => ["_grokparsefailure_request"]
    }
    
    # 日期解析
    date {
      match => ["timestamp", "yyyy-MM-dd HH:mm:ss.SSS", "yyyy-MM-dd'T'HH:mm:ss.SSS'Z'", "ISO8601"]
      target => "@timestamp"
      timezone => "Asia/Shanghai"
    }
    
    # GeoIP 解析（如果有 IP）
    if [client_ip] {
      geoip {
        source => "client_ip"
        target => "geoip"
        fields => ["country_name", "region_name", "city_name", "latitude", "longitude"]
      }
    }
  }
  
  # ===== 访问日志处理 =====
  if [topic] == "access-logs" {
    # 解析 Nginx 访问日志
    grok {
      match => { "message" => "%{IPORHOST:remote_addr} - %{DATA:remote_user} \[%{HTTPDATE:time_local}\] \"%{WORD:method} %{DATA:request_uri} HTTP/%{NUMBER:http_version}\" %{NUMBER:status} %{NUMBER:body_bytes_sent} \"%{DATA:http_referer}\" \"%{DATA:http_user_agent}\" \"%{DATA:x_forwarded_for}\" %{NUMBER:request_time}" }
      tag_on_failure => ["_grokparsefailure_access"]
    }
    
    # 日期解析
    date {
      match => ["time_local", "dd/MMM/yyyy:HH:mm:ss Z"]
      target => "@timestamp"
      timezone => "Asia/Shanghai"
    }
    
    # 类型转换
    mutate {
      convert => {
        "status" => "integer"
        "body_bytes_sent" => "integer"
        "request_time" => "float"
      }
    }
    
    # GeoIP
    geoip {
      source => "remote_addr"
      target => "geoip"
      fields => ["country_name", "region_name", "city_name"]
    }
  }
  
  # ===== 系统日志处理 =====
  if [topic] == "system-logs" {
    # 解析 syslog 格式
    grok {
      match => { "message" => "%{SYSLOGTIMESTAMP:syslog_timestamp} %{SYSLOGHOST:hostname} %{DATA:program}(?:\[%{POSINT:pid}\])?: %{GREEDYDATA:syslog_message}" }
      tag_on_failure => ["_grokparsefailure_syslog"]
    }
  }
  
  # ===== 审计日志处理 =====
  if [topic] == "audit-logs" {
    # 审计日志保持原始格式
    mutate {
      add_field => {
        "log_type" => "audit"
      }
    }
  }
  
  # ===== 字段标准化 =====
  mutate {
    rename => {
      "log_level" => "level"
      "logstash_processed_at" => "processed_at"
    }
    add_field => {
      "log_source" => "logstash"
    }
  }
}

output {
  # ===== 应用日志输出 =====
  if [topic] == "app-logs" {
    elasticsearch {
      hosts => ["elasticsearch-master-0:9200", "elasticsearch-master-1:9200", "elasticsearch-master-2:9200"]
      index => "app-logs-%{+YYYY.MM.dd}"
      user => "elastic"
      password => "${ES_PASSWORD}"
      ssl => true
      cacert => "/etc/logstash/certs/ca.crt"
      
      # ILM 配置
      ilm_enabled => true
```

### 9.1 索引管理脚本

```bash
#!/bin/bash
# es-index-manager.sh - ES 索引管理工具

set -euo pipefail

ES_URL="https://elasticsearch-master:9200"
ES_USER="elastic"
ES_PASS="ElasticAdmin@2026!"

log() { echo "[$(date '+%H:%M:%S')] $1"; }

# 查看索引信息
show_indices() {
  log "=== ES 索引信息 ==="
  curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/_cat/indices?v&s=index" | \
    awk '{printf "%-40s %-10s %-10s %-10s\n", $1, $2, $3, $4}'
}

# 删除过期索引
cleanup_indices() {
  local days=${1:-30}
  log "=== 删除 ${days} 天前的索引 ==="
  
  # 计算日期
  cutoff_date=$(date -d "-${days} days" +%Y.%m.%d)
  
  # 获取需要删除的索引
  indices=$(curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/_cat/indices?v&h=index" | \
    grep -E "^app-logs-|^access-logs-|^system-logs-" | \
    while read idx; do
      idx_date=$(echo ${idx} | grep -oP '\d{4}\.\d{2}\.\d{2}')
      if [[ "${idx_date}" < "${cutoff_date}" ]]; then
        echo ${idx}
      fi
    done)
  
  # 删除索引
  for idx in ${indices}; do
    log "删除索引: ${idx}"
    curl -sk -u "${ES_USER}:${ES_PASS}" -X DELETE "${ES_URL}/${idx}"
  done
  
  log "清理完成"
}

# 索引健康检查
health_check() {
  log "=== ES 集群健康检查 ==="
  curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/_cluster/health?pretty"
  
  log "=== 节点状态 ==="
  curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/_cat/nodes?v"
  
  log "=== 磁盘使用 ==="
  curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/_cat/allocation?v"
}

# 主菜单
echo "选择操作:"
echo "1. 查看索引"
echo "2. 清理索引"
echo "3. 健康检查"

read -p "选择: " choice
case $choice in
  1) show_indices ;;
  2) read -p "保留天数 (默认30): " days; cleanup_indices ${days:-30} ;;
  3) health_check ;;
  *) echo "无效选择" ;;
esac
```

```bash
#!/bin/bash
# es-capacity-prediction.sh - ES 容量预测

set -euo pipefail

ES_USER="elastic"

  size=$(curl -sk -u "${ES_USER}:${ES_PASS}" "${ES_URL}/app-logs-${date}/_stats/store" | \
    jq '.indices."app-logs-'${date}'".total.store.size_in_bytes // 0')
```

## 十、面试题与思考

### Q1: 为什么要在 Logstash 和 Elasticsearch 之间加 Kafka？

**答：** 加 Kafka 作为缓冲层有以下好处：

1. **解耦**：日志采集（Filebeat）和日志处理（Logstash）解耦，任一组件故障不影响其他组件
2. **削峰填谷**：日志高峰期（如促销活动），Kafka 可以缓冲大量数据，避免 ES 被压垮
3. **可靠性**：Kafka 保证消息不丢失，即使 Logstash 宕机，日志也不会丢失
4. **扩展性**：可以动态增加 Logstash 消费者来处理更多日志
5. **多消费者**：同一份日志可以被多个系统消费（ES、HDFS、实时分析）

### Q2: ELK 如何保证日志不丢失？

**答：** 从四个层面保证：

1. **Filebeat → Kafka**：使用 ACK 机制，确保数据写入 Kafka
2. **Kafka → Logstash**：使用 consumer offset，Logstash 处理完成后才提交 offset
3. **Logstash → ES**：使用 `refresh_interval` 和 `translog`，确保数据持久化
4. **ES 持久化**：使用副本和分片，保证数据冗余

**关键配置：**
- Filebeat: `acknowledgement: true`
- Kafka: `min.insync.replicas: 2`
- Logstash: `flush_size` 和 `idle_flush_time`
- ES: `index.translog.durability: async`

### Q3: 如何优化 ES 查询性能？

**答：**

1. **索引优化**：
   - 合理设置分片数（每个分片 10-50GB）
   - 使用 ILM 管理索引生命周期
   - 冷热分离，热数据用 SSD

2. **查询优化**：
   - 使用 filter 代替 query（filter 可缓存）
   - 避免使用 wildcard 和正则
   - 使用 `track_total_hits: false` 避免精确计数
   - 使用 `search_after` 代替 `from+size` 深度分页

3. **映射优化**：
   - 使用 `keyword` 代替 `text`（如果不需要分词）
   - 禁用不需要搜索的字段的索引
   - 使用 `doc_values` 加速排序

4. **硬件优化**：
   - 使用 SSD
   - 增加内存（JVM 堆不超过 32GB）
   - 使用多磁盘分摊 I/O

### Q4: Filebeat 的背压(backpressure)机制是什么？

**答：** 背压是 Filebeat 处理生产者速度过快时的保护机制：

1. **检测背压**：当 Kafka 消费者处理不过来时，Filebeat 会检测到
2. **减速**：Filebeat 自动降低采集速度
3. **通知**：日志中会输出 "Connection to backoff" 消息
4. **恢复**：当消费者处理能力恢复时，Filebeat 自动恢复正常速度

**配置参数：**
```yaml
queue.mem:
  events: 4096
  flush.min_events: 512
  flush.timeout: 5s
```

### Q5: 如何设计日志的多租户隔离？

**答：** 多租户日志隔离方案：

1. **索引级别隔离**：每个租户独立索引，如 `tenant-a-logs-*`、`tenant-b-logs-*`
2. **命名空间隔离**：使用 K8s Namespace 隔离不同租户的日志采集
3. **RBAC 隔离**：ES RBAC 控制不同用户只能访问对应租户的索引
4. **Kibana 空间隔离**：使用 Kibana Spaces 隔离不同租户的仪表盘
5. **资源隔离**：为每个租户配置独立的 Logstash pipeline 和 Kafka topic
