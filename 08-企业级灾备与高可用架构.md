## 二、多AZ部署配置

### 2.1 多AZ Pod 反亲和性

```yaml
# multi-az-pod.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-multi-az
  namespace: production
spec:
  replicas: 6
  selector:
    matchLabels:
      app: app-multi-az
  template:
    metadata:
      labels:
        app: app-multi-az
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: app-multi-az
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels:
              app: app-multi-az
      
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchExpressions:
                    - key: app
                      operator: In
                      values:
                        - app-multi-az
                topologyKey: kubernetes.io/hostname
      
      containers:
        - name: app
          image: harbor.example.com/production/app:v1.0.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              cpu: "1"
              memory: "1Gi"
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20
```

## 五、高可用脚本

```bash
#!/bin/bash
# failover.sh - 故障切换脚本

set -euo pipefail

echo "=== 故障切换 ==="

# 1. 检查主集群状态
echo "检查主集群状态..."
MASTER_STATUS=$(curl -sk https://master.example.com/health | jq -r '.status')

if [ "${MASTER_STATUS}" = "healthy" ]; then
  echo "主集群正常，无需切换"
  exit 0
fi

# 2. 切换 DNS 到灾备集群
echo "切换 DNS 到灾备集群..."
# 更新 DNS 记录指向灾备集群
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890 \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "TTL": 60,
        "ResourceRecords": [{"Value": "10.1.0.100"}]
      }
    }]
  }'

# 3. 通知相关人员
echo "发送告警通知..."
curl -X POST https://oapi.dingtalk.com/robot/send?access_token=xxx \
  -H 'Content-Type: application/json' \
  -d '{
    "msgtype": "text",
    "text": {
      "content": "⚠️ 故障切换已执行！DNS 已切换到灾备集群"
    }
  }'

echo "=== 故障切换完成 ==="
```

## 六、面试题与思考

### Q1: RPO 和 RTO 是什么？如何优化？

**答：**

- **RPO (Recovery Point Objective)**：可接受的数据丢失量。RPO=5min 表示最多丢失 5 分钟数据
- **RTO (Recovery Time Objective)**：可接受的服务中断时间。RTO=30min 表示最多中断 30 分钟

**优化策略：**
1. **降低 RPO**：同步复制、实时备份、减少备份间隔
2. **降低 RTO**：自动化故障切换、预热环境、健康检查
3. **权衡**：RPO 和 RTO 越小，成本越高

### Q2: 同城双活和异地灾备有什么区别？

**答：**

| 特性 | 同城双活 | 异地灾备 |
|------|---------|---------|
| 距离 | < 100km | > 500km |
| 延迟 | < 1ms | > 10ms |
| 数据同步 | 同步 | 异步 |
| RPO | 接近 0 | 分钟级 |
| RTO | 秒级 | 分钟级 |
| 成本 | 中等 | 高 |
| 用途 | 高可用 | 灾难恢复 |

### Q3: 混沌工程的核心原则是什么？

**答：**

1. **建立假设**：定义系统在正常状态下的行为
2. **设计实验**：模拟真实世界的故障场景
3. **执行实验**：在生产环境（或预生产）执行实验
4. **分析结果**：对比实验结果与假设
5. **改进系统**：根据结果修复问题
6. **自动化**：将实验自动化，持续验证
