### 1.2 节点规划

| 角色 | 数量 | 规格 | 磁盘 | 用途 |
|------|------|------|------|------|
| MON | 3 | 4C8G | 100GB SSD | 集群监控仲裁 |
| MGR | 2 | 4C8G | 100GB SSD | 管理面板 |
| MDS | 2 | 4C8G | 100GB SSD | CephFS 元数据 |
| OSD | 6+ | 8C16G | 1TB SSD x2 | 数据存储 |
| RGW | 2 | 4C8G | 100GB SSD | 对象网关 |

### 1.3 网络规划

```
Public Network (客户端通信): 10.0.100.0/24
  - MON: 10.0.100.11-13
  - MGR: 10.0.100.21-22
  - OSD: 10.0.100.31-36
  
Cluster Network (OSD 复制/心跳): 10.0.200.0/24
  - 专用网络，避免客户端流量影响复制性能
  
Client Network (应用访问): 10.0.0.0/24
  - K8s Pod 通过 CSI 驱动访问
```

## 五、PG 数量计算

### 5.1 PG 计算公式

```
Total PGs = (OSDs * 100) / Replicas

建议值:
- 最小 PG 数: 128
- 最大 PG 数: 32768
- 每个 OSD 目标 PG 数: 100
- PG 自动调整: 启用
```

### 5.2 PG 自动调整配置

```bash
#!/bin/bash
# pg-autoscaler-config.sh - 配置 PG 自动调整

set -euo pipefail

CEPH_CMD="kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- ceph"

echo "=== 配置 PG 自动调整 ==="

# 启用 pg_autoscaler 模块
${CEPH_CMD} mgr module enable pg_autoscaler

# 设置默认自动调整模式
${CEPH_CMD} config set global pg_autoscaler_mode on

# 设置目标比例
${CEPH_CMD} config set global osd_pool_default_target_size_ratio 0.5

# 查看当前 PG 状态
echo "--- 当前 PG 状态 ---"
${CEPH_CMD} osd pool stats

# 查看自动调整建议
echo "--- 自动调整建议 ---"
${CEPH_CMD} osd pool autoscaler status

echo "=== 配置完成 ==="
```

## 六、Ceph 故障处理

### 6.1 OSD 故障处理

```bash
#!/bin/bash
# osd-failure-handler.sh - OSD 故障自动处理

set -euo pipefail

CEPH_CMD="kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- ceph"

log() { echo "[$(date '+%H:%M:%S')] $1"; }

# 检查 OSD 状态
check_osd_status() {
  log "=== 检查 OSD 状态 ==="
  ${CEPH_CMD} osd tree
  echo ""
  ${CEPH_CMD} health detail
}

# 处理 OSD 故障
handle_osd_failure() {
  local osd_id=${1:?需要 OSD ID}
  
  log "=== 处理 OSD ${osd_id} 故障 ==="
  
  # 标记 OSD 为 down
  log "标记 OSD ${osd_id} 为 down..."
  ${CEPH_CMD} osd down ${osd_id}
  
  # 标记 OSD 为 out（从 CRUSH 树移除）
  log "标记 OSD ${osd_id} 为 out..."
  ${CEPH_CMD} osd out ${osd_id}
  
  # 等待数据重平衡
  log "等待数据重平衡..."
  ${CEPH_CMD} -w
  
  # 查看重平衡状态
  log "重平衡状态:"
  ${CEPH_CMD} osd reweight-tree
}

# 替换 OSD
replace_osd() {
  local osd_id=${1:?需要 OSD ID}
  local node_name=${2:?需要节点名称}
  
  log "=== 替换 OSD ${osd_id} ==="
  
  # 停止 OSD
  ${CEPH_CMD} osd stop ${osd_id}
  
  # 从 CRUSH 树移除
  ${CEPH_CMD} osd crush remove osd.${osd_id}
  
  # 删除 OSD 密钥
  ${CEPH_CMD} auth del osd.${osd_id}
  
  # 删除 OSD
  ${CEPH_CMD} osd rm ${osd_id}
  
  # 清理旧磁盘
  log "清理旧磁盘..."
  kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-osd-${osd_id} -o jsonpath='{.items[0].metadata.name}' 2>/dev/null || echo "none") -- \
    zap磁盘 || echo "无法清理，手动处理"
  
  log "OSD ${osd_id} 已移除，请在新节点上创建新 OSD"
}

# 主菜单
echo "OSD 故障处理工具"
echo "1. 检查 OSD 状态"
echo "2. 处理 OSD 故障"
echo "3. 替换 OSD"

read -p "选择: " choice
case $choice in
  1) check_osd_status ;;
  2) read -p "OSD ID: " osd_id; handle_osd_failure ${osd_id} ;;
  3) read -p "OSD ID: " osd_id; read -p "节点名称: " node; replace_osd ${osd_id} ${node} ;;
  *) echo "无效选择" ;;
esac
```

### 6.2 数据恢复流程

```bash
#!/bin/bash
# data-recovery.sh - Ceph 数据恢复流程

set -euo pipefail

CEPH_CMD="kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- ceph"

log() { echo "[$(date '+%H:%M:%S')] $1"; }

# 检查数据一致性
check_data_consistency() {
  log "=== 检查数据一致性 ==="
  
  # 检查 PG 状态
  ${CEPH_CMD} pg stat
  
  # 检查不一致的 PG
  inconsistent_pgs=$(${CEPH_CMD} pg dump pgs_brief | grep "inconsistent" | wc -l)
  
  if [ "${inconsistent_pgs}" -gt 0 ]; then
    log "⚠️ 发现 ${inconsistent_pgs} 个不一致的 PG"
    ${CEPH_CMD} pg dump pgs_brief | grep "inconsistent"
  else
    log "✅ 数据一致性检查通过"
  fi
}

# 修复不一致的 PG
repair_inconsistent_pgs() {
  log "=== 修复不一致的 PG ==="
  
  # 获取不一致的 PG 列表
  pgs=$(${CEPH_CMD} pg dump pgs_brief | grep "inconsistent" | awk '{print $1}')
  
  for pg in ${pgs}; do
    log "修复 PG: ${pg}"
    ${CEPH_CMD} pg repair ${pg}
  done
  
  log "修复完成，等待数据同步..."
  sleep 60
  
  # 验证修复结果
  check_data_consistency
}

# 主菜单
echo "数据恢复工具"
echo "1. 检查数据一致性"
echo "2. 修复不一致的 PG"

read -p "选择: " choice
case $choice in
  1) check_data_consistency ;;
  2) repair_inconsistent_pgs ;;
  *) echo "无效选择" ;;
esac
```

## 十一、面试题与思考

### Q1: Ceph 的 CRUSH 算法是什么？如何工作？

**答：** CRUSH (Controlled Replication Under Scalable Hashing) 是 Ceph 的数据分布算法：

**工作原理：**
1. **输入**：数据对象 ID + CRUSH Map + CRUSH 规则
2. **计算**：根据对象 ID 计算哈希值，映射到 PG
3. **选择**：根据 CRUSH 规则，选择 OSD 列表
4. **分布**：数据按权重分布在多个 OSD 上

**优势：**
- **去中心化**：不需要中央查找表
- **可扩展**：支持数千 OSD
- **故障域**：支持机架、主机、机柜等多级故障域
- **权重**：根据 OSD 容量设置权重

**示例：**
```
# 3 副本，故障域为 host
rule replicated_rule {
    ruleset 0
    type replicated
    min_size 1
    max_size 10
    step take default class ssd
    step choose firstn 0 type host
    step emit
}
```

### Q2: Rook 和 Longhorn 如何选择？

**答：** 选择依据：

| 特性 | Rook Ceph | Longhorn |
|------|-----------|----------|
| 复杂度 | 高（学习曲线陡） | 低（简单易用） |
| 性能 | 高（生产级） | 中等 |
| 功能 | 块/文件/对象存储 | 块存储 |
| 适用场景 | 大规模、多协议 | 中小规模、简单需求 |
| 资源消耗 | 高（每 OSD 1-2GB 内存） | 低 |
| 数据保护 | 副本 + 纠删码 | 副本 |
| 快照 | 支持 | 支持 |
| 备份 | 支持 | 支持 |

**选择建议：**
- **选 Rook Ceph**：需要对象存储、大规模集群（>10 节点）、高性能
- **选 Longhorn**：中小规模集群、快速部署、简单需求

### Q3: 如何保证存储的高可用？

**答：** 从多个层面保证：

1. **副本策略**：
   - 至少 3 副本，分布在不同故障域
   - `minSize: 2` 保证部分 OSD 故障时仍可写入

2. **故障域隔离**：
   - CRUSH 规则指定故障域（host/rack/zone）
   - 数据分布在不同物理位置

3. **自动修复**：
   - OSD 故障时自动重平衡
   - PG 不一致时自动修复

4. **监控告警**：
   - 监控 OSD 状态、磁盘健康
   - PG 不干净告警

5. **备份恢复**：
   - 定期快照备份
   - 跨集群复制

### Q4: Ceph 的三种存储类型有什么区别？

**答：**

| 类型 | RBD | CephFS | RGW |
|------|-----|--------|-----|
| 协议 | 块设备 | POSIX 文件系统 | S3/HTTP |
| 访问方式 | 挂载为块设备 | 挂载为文件系统 | HTTP API |
| 性能 | 最高 | 中等 | 中等 |
| 适用场景 | 数据库、虚拟机 | 共享文件、日志 | 备份、归档 |
| 读写模式 | 独占读写 | 多客户端读写 | 多客户端读写 |
| K8s 集成 | PVC | PVC | CSI Driver |

**使用建议：**
- **数据库/有状态应用** → RBD（高性能、独占）
- **共享文件/日志** → CephFS（多节点读写）
- **备份/归档** → RGW（对象存储、低成本）

### Q5: 如何进行 Ceph 的容量规划？

**答：** 容量规划公式：

```
所需容量 = 数据量 × 副本数 × 冗余系数

示例：
- 原始数据: 1TB
- 副本数: 3
- 冗余系数: 1.2 (20% 冗余)
- 所需容量: 1TB × 3 × 1.2 = 3.6TB

OSD 数量规划：
- 每个 OSD 容量: 1TB
- 所需 OSD 数: 3.6TB / 1TB = 4 个
- 建议: 6 个 OSD（提供更好容错）
```

**注意事项：**
- 预留 20-30% 空间用于重平衡
- OSD 利用率不超过 80%
- 定期检查容量趋势
