## 二、Rook Ceph 部署

### 2.1 Rook Operator 部署

```yaml
# rook-ceph-operator.yaml
apiVersion: rook.io/v1
kind: CephCluster
metadata:
  name: rook-ceph
  namespace: rook-ceph
spec:
  cephVersion:
    image: ceph/ceph:v18.2.1
    allowUnsupported: false
  
  dataDirHostPath: /var/lib/rook
  
  mon:
    count: 3
    allowMultiplePerNode: false
  
  mgr:
    count: 2
    modules:
      - name: pg_autoscaler
        enabled: true
      - name: rook
        enabled: true
  
  dashboard:
    enabled: true
    ssl: true
  
  monitoring:
    enabled: true
  
  storage:
    useAllNodes: true
    useAllDevices: true
    deviceFilter: "sd.*"
    config:
      databaseSizeMB: "10240"
      journalSizeMB: "10240"
  
  resources:
    mgr:
      requests:
        cpu: 500m
        memory: 512Mi
      limits:
        cpu: "1"
        memory: 1Gi
    mon:
      requests:
        cpu: 500m
        memory: 1Gi
      limits:
        cpu: "1"
        memory: 2Gi
    osd:
      requests:
        cpu: "1"
        memory: 2Gi
      limits:
        cpu: "2"
        memory: 4Gi
  
  healthCheck:
    daemonHealth:
      mon:
        interval: 60s
      osd:
        interval: 60s
      mgr:
        interval: 60s
  
  disruptionManagement:
    managePodBudgets: true
    osdMaintenanceTimeout: 300
```

### 2.2 StorageClass 配置

```yaml
# ceph-storageclass.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ceph-rbd
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: rook-ceph.rbd.csi.ceph.com
parameters:
  clusterID: rook-ceph
  pool: replicapool
  imageFeatures: layering
  csi.storage.k8s.io/provisioner-secret-name: rook-csi-rbd-provisioner
  csi.storage.k8s.io/provisioner-secret-namespace: rook-ceph
  csi.storage.k8s.io/node-stage-secret-name: rook-csi-rbd-node
  csi.storage.k8s.io/node-stage-secret-namespace: rook-ceph
  csi.storage.k8s.io/fstype: ext4
reclaimPolicy: Delete
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer

---
# CephFS StorageClass
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: cephfs
provisioner: rook-cephfs.csi.ceph.com
parameters:
  clusterID: rook-ceph
  fsName: myfs
  pool: myfs-data0
  csi.storage.k8s.io/provisioner-secret-name: rook-csi-cephfs-provisioner
  csi.storage.k8s.io/provisioner-secret-namespace: rook-ceph
  csi.storage.k8s.io/node-stage-secret-name: rook-csi-cephfs-node
  csi.storage.k8s.io/node-stage-secret-namespace: rook-ceph
reclaimPolicy: Delete
allowVolumeExpansion: true
```

## 四、PVC 示例

```yaml
# pvc-example.yaml
# 块存储 PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-data
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ceph-rbd
  resources:
    requests:
      storage: 100Gi

---
# 文件存储 PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-data
  namespace: production
spec:
  accessModes:
    - ReadWriteMany
  storageClassName: cephfs
  resources:
    requests:
      storage: 50Gi

---
# StatefulSet 使用 PVC
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
  namespace: production
spec:
  serviceName: mysql
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
          resources:
            requests:
              cpu: "1"
              memory: 2Gi
            limits:
              cpu: "2"
              memory: 4Gi
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: ceph-rbd
        resources:
          requests:
            storage: 100Gi
```

## 五、运维脚本

```bash
#!/bin/bash
# ceph-health-check.sh - Ceph 健康检查

set -euo pipefail

echo "=== Ceph 集群健康检查 ==="

# 集群状态
echo "--- 集群状态 ---"
kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- \
  ceph health detail

# OSD 状态
echo ""
echo "--- OSD 状态 ---"
kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- \
  ceph osd tree

# 存储使用
echo ""
echo "--- 存储使用 ---"
kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- \
  ceph df

# 坏块检查
echo ""
echo "--- 坏块检查 ---"
kubectl exec -n rook-ceph $(kubectl get pod -n rook-ceph -l app=rook-ceph-mon -o jsonpath='{.items[0].metadata.name}') -- \
  ceph health detail | grep -i "slow\|stale\|degraded"

echo ""
echo "=== 检查完成 ==="
```

## 六、面试题与思考

### Q1: Ceph 的 CRUSH 算法是什么？

**答：** CRUSH (Controlled Replication Under Scalable Hashing) 是 Ceph 的数据分布算法：

1. **原理**：根据存储权重和故障域，计算数据应该存储在哪些 OSD 上
2. **优势**：去中心化、无单点故障、可扩展
3. **故障域**：支持机架、主机、机柜等多级故障域
4. **权重**：根据 OSD 容量设置权重，数据按权重分布

### Q2: Rook 和 Longhorn 如何选择？

**答：** 选择依据：

| 特性 | Rook Ceph | Longhorn |
|------|-----------|----------|
| 复杂度 | 高（学习曲线陡） | 低（简单易用） |
| 性能 | 高（生产级） | 中等 |
| 功能 | 块/文件/对象存储 | 块存储 |
| 适用场景 | 大规模、多协议 | 中小规模、简单需求 |
| 资源消耗 | 高 | 低 |

### Q3: 如何保证存储的高可用？

**答：** 从多个层面保证：

1. **副本策略**：至少 3 副本，分布在不同故障域
2. **纠删码**：使用纠删码提高存储效率
3. **快照备份**：定期快照备份到远程存储
4. **监控告警**：监控 OSD 状态、磁盘健康
5. **自动修复**：Ceph 自动检测和修复数据不一致
